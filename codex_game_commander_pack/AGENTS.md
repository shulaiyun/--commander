# AGENTS.md

## 项目身份

本仓库是一个近现代战争题材的 **沙盒策略 RPG / 战役经营游戏**。
项目代号：**裂变纪元**

目标不是做纯大战略，也不是纯动作游戏，而是做一个介于两者之间、强调：
- 自由角色成长
- 势力加入 / 背叛 / 自立
- 大地图推进
- 关键节点占领
- 海陆空多层战斗
- 历史事件驱动
- 轻社交协作

## 当前技术基线

- 引擎：Godot
- 语言：GDScript 优先
- 架构原则：数据驱动、模块化、低耦合
- 视角：2.5D / 俯视角
- 目标平台：PC 优先，后续再看移动端

## 你在这个仓库中的工作方式

每次开始工作前，都必须先做下面 5 件事：

1. 阅读本文件 `AGENTS.md`
2. 阅读 `PLANS.md`
3. 阅读本次任务相关文档
4. 复述你理解的任务边界
5. 只实现本次任务，不擅自扩展范围

## 禁止行为

- 不要把项目扩成 MMO
- 不要擅自增加大量外部依赖
- 不要引入与当前里程碑无关的复杂联网系统
- 不要把单个任务扩展成“顺手重构全项目”
- 不要在没有明确要求时改动美术风格
- 不要写真实政治宣传内容
- 不要把近现代敏感历史事件做成猎奇或娱乐化暴行模拟
- 不要在没有验证的前提下声称“功能完成”

## 当前产品原则

### 产品支柱
1. 占领关键地点要有成就感
2. 战争推进要受补给和工业约束
3. 角色成长要能影响战局
4. 历史事件既提供厚重感，也允许有限改写
5. 社交必须服务留存，而不是破坏单机核心

### 体验优先级
1. 可玩性
2. 清晰反馈
3. 稳定性
4. 可扩展性
5. 视觉 polish

## 当前首发范围（MVP）

只做：
- 一个世界地图垂直切片
- 3~4 个可交互势力
- 8~12 个关键城市/港口/节点
- 基础人物系统
- 基础任务系统
- 占领 / 补给 / 事件
- 1 种陆战玩法
- 海战 / 空战先做最小可验证版本
- 轻社交原型（异步）

不做：
- 全球全时代完整内容
- 大规模实时多人同屏
- 复杂经济市场模拟
- 高复杂度真实外交条约系统
- 完整冷战与现代章节

## 输出要求

每次完成任务时，必须输出：

1. 你改了哪些文件
2. 为什么这么改
3. 如何运行 / 如何验证
4. 还剩哪些已知问题
5. 下一步最合理的最小任务是什么

## “完成”的定义

一个任务只有在以下条件满足时，才算完成：

- 功能实现
- 没有明显报错
- 能运行或至少能被本地验证
- 更新了相关文档
- 没有无关范围漂移
- 给出清晰的测试方式

## 代码风格

- 优先清晰命名
- 优先小模块
- 优先数据配置而不是硬编码
- 尽量让系统能被设计表驱动
- 写必要注释，但不要写噪音注释
- 不要过度抽象
- 先保证一条简单路径可用，再扩展

## 数据设计要求

以下内容尽量使用 JSON / Resource / 配置文件表达，而不是散落在脚本里：

- 势力
- 城市
- 将领 / 人物
- 装备
- 战役事件
- 科技节点
- 社交配置
- 地图点位

## 社交系统规则

社交功能默认按以下顺序推进：

### 第一阶段：轻社交
- 排行榜
- 战报分享
- 档案页
- 战绩勋章
- 异步排行榜奖励

### 第二阶段：弱协作
- 军团
- 阵营聊天（占位实现即可）
- 联合任务
- 协作目标进度

### 第三阶段：强协作
只有在单机核心完成、数据留存指标明确后，才允许推进：
- 异步攻城
- 跨服事件
- 赛季地图

### 第四阶段：实时联机
默认不做，除非单独立项。

## 每次做复杂任务时

复杂任务必须先产出：
- 任务理解
- 风险列表
- 分步计划
- 验收标准

如果任务模糊，先收口，不要直接开写。

## 建议的任务拆法

优先按下面粒度拆任务：
- 一个 UI 面板
- 一个数据模型
- 一个系统闭环
- 一个垂直切片
- 一个验证脚本

不要一次覆盖多个大系统。
